# Implementation Plan: Signup Application

This document outlines the technical implementation plan for the Signup Application based on the [Functional Requirements](./functional_requirements.md).

## 1. Technology Stack & Prerequisites

- **Frontend/Backend**: Next.js (App Router)
- **Authentication**: BetterAuth with Discord OAuth2 provider
- **Database**: PostgreSQL
- **ORM/Query Builder**: Drizzle ORM (recommended for type safety and performance)
- **Styling**: Tailwind CSS
- **Time Management**: `date-fns` or `dayjs` for timezone and slot calculations

### Prerequisites
- Discord Developer Application (Client ID & Secret)
- PostgreSQL database instance
- Configured environment variables (`BETTER_AUTH_SECRET`, `DISCORD_CLIENT_ID`, `DISCORD_CLIENT_SECRET`, `DATABASE_URL`)

---

## 2. Phase 1: Environment & Database Setup

1. **Database Schema Definition**:
    - **Users Table**: `id`, `name`, `email`, `image`, `discord_id`, `roles`.
    - **Sessions Table**: Required by BetterAuth.
    - **Accounts Table**: Required by BetterAuth for OAuth linking.
    - **Slots Table**:
        - `id` (UUID)
        - `user_id` (FK to Users, nullable)
        - `start_time` (Timestamp with timezone, unique index)
        - `duration_minutes` (Integer: 60 for 1h, 120 for 2h)
        - `is_blocked` (Boolean, for admin overrides)

2. **Drizzle ORM Configuration**:
    - Initialize Drizzle with PostgreSQL driver (`postgres` or `pg`).
    - Generate and run initial migrations.

---

## 3. Phase 2: Authentication & Authorization

1. **BetterAuth Integration**:
    - Configure `better-auth` client and server.
    - Set up Discord provider.
    - Implement session validation middleware/hooks.
2. **Discord Role Mapping**:
    - Implement logic to fetch guild roles during/after OAuth callback.
    - Create a utility to map Discord role IDs to application permissions (`is_admin`, `can_claim`).

---

## 4. Phase 3: Core API Routes

1. **`GET /api/slots?start=...&end=...`**:
    - Fetch all slots within the requested 7-day range.
    - Merge database-assigned slots with "virtual" available slots generated by the schedule logic.
2. **`POST /api/slots/claim`**:
    - Input: `start_time`, `duration`.
    - Check if slot is already taken (Server-side validation).
    - Insert record into `Slots` table.
3. **`POST /api/slots/release`**:
    - Input: `slot_id`.
    - Authorization check: User must own the slot OR be an Admin.
    - Delete record from `Slots` table.

---

## 5. Phase 4: Frontend Development

1. **Schedule Grid Logic**:
    - Develop a utility function to generate the 7-day rolling window starting from `now()`.
    - Logic for variable rows:
        - 12 AM - 4 PM: 1-hour intervals.
        - 4 PM - 12 AM: 2-hour intervals (single blocks).
    - Convert all UTC timestamps from API to user's local time using `Intl.DateTimeFormat`.
2. **Components**:
    - `ScheduleTable`: Main grid container.
    - `DayColumn`: Individual columns for the 7-day view.
    - `TimeSlot`: Clickable cell with status states (Available, Claimed, Occupied).
3. **State Management & Optimistic UI**:
    - Use React Query (TanStack Query) for data fetching and mutations.
    - Implement optimistic updates for "Claim" and "Release" actions to ensure a snappy user experience.

---

## 6. Phase 5: Admin Features

1. **Admin Override UI**:
    - Detect admin role via session.
    - Enable "Release" button on any occupied slot for admin users.
2. **Admin View**:
    - Optional: A simplified dashboard to see all upcoming assignments in a list view.

---

## 7. Phase 6: Testing & Deployment

1. **Unit Testing**:
    - Test slot generation logic (handling the 1h/2h split).
    - Test timezone conversion utilities.
2. **Integration Testing**:
    - Test claim/release flow with mock database.
3. **Deployment**:
    - Deploy to Vercel or similar.
    - Configure production database and Discord redirect URIs.

---

## 8. Critical Considerations

- **Race Conditions**: Use database transactions or `UNIQUE` constraints on `start_time` in the `Slots` table to prevent double-claiming.
- **Timezone Edge Cases**: Ensure the "rolling 7 days" starts at the beginning of the user's *current day* in their local time, but queries the database using UTC ranges.
- **Discord API Rate Limits**: Cache user roles in the session to avoid hitting Discord API on every page load.
